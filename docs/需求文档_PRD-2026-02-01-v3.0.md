# 金融案件证据集生成系统 v3.0 - 产品需求文档

**文档编号**: PRD-2026-02-01-v3.0
**版本**: v3.0.0
**日期**: 2026-02-01

---

## 一、产品概述

### 1.1 产品目标

输入一份判决书PDF，输出一份符合法院标准的证据集PDF。

### 1.2 用户故事

| 角色 | 场景 | 需求 |
|------|------|------|
| 法官 | 收到金融案件判决书（融资租赁/借款/保理等） | 快速生成完整的证据提交材料 |
| 律师 | 代理金融案件 | 整理证据材料，生成标准化的证据集 |
| 当事人 | 需要提交证据材料 | 一键生成符合法院要求的证据文档 |

### 1.4 案件类型支持

本系统是一个**通用证据生成框架**，支持多种金融案件类型。

| 案件类型 | 核心证据 | 典型附件 |
|----------|----------|----------|
| 融资租赁纠纷 | 融资租赁合同 | 租赁物清单、租金支付计划 |
| 金融借款合同纠纷 | 借款合同 | 还款计划、抵押物清单 |
| 民间借贷纠纷 | 借条/借款合同 | 还款记录 |
| 保理合同纠纷 | 保理合同 | 应收账款清单、转让通知 |
| 担保合同纠纷 | 担保合同 | 担保物清单 |

> **扩展说明**：新增案件类型只需配置证据类型和附件类型，无需修改系统代码。

### 1.3 产品愿景

**告诉系统要什么，系统就生成什么。**

## 二、系统边界

### 2.1 输入输出

```
输入: 判决书 PDF
输出: 证据集 PDF（包含起诉状 + 证据材料）
```

### 2.2 功能范围

| 功能 | 说明 |
|------|------|
| 判决书解析 | 理解判决书内容，提取关键信息 |
| 证据生成 | 根据判决书内容生成证据材料 |
| PDF输出 | 生成符合法院标准的证据集 |

### 2.3 非功能范围（不包含）

- 判决书内容生成
- 法院系统对接
- 电子送达

## 三、功能清单

### 3.0 案件类型与证据类型配置

本系统通过配置化设计支持多种案件类型。系统流程如下：

```
用户选择案件类型
     ↓
根据案件类型确定需要生成哪些证据
     ↓
根据证据类型确定需要提取哪些信息
     ↓
从判决书中提取这些信息
     ↓
生成证据
```

#### 3.0.1 案件类型与证据类型映射

| 案件类型 | 需要生成的证据 |
|----------|----------------|
| 融资租赁纠纷 | 融资租赁合同、付款凭证、公证书、租赁物清单、租金支付计划 |
| 金融借款合同纠纷 | 借款合同、银行流水、还款计划、担保合同 |
| 民间借贷纠纷 | 借条、银行转账凭证、还款记录 |
| 保理合同纠纷 | 保理合同、应收账款清单、转让通知 |
| 担保合同纠纷 | 担保合同、抵押物清单、他项权利证明 |

#### 3.0.2 证据类型与信息需求映射

| 证据类型 | 需要的信息 |
|----------|------------|
| 合同类 | 当事人信息、标的物描述、金额条款、日期、签署栏 |
| 凭证类 | 金额、日期、收付款方、交易摘要 |
| 文书类 | 文书类型、发证机关、关键内容、日期 |
| 附件类 | 序号、名称、数量/金额（表格形式） |

> **扩展说明**：新增案件类型只需配置对应的证据类型列表；新增证据类型只需配置需要提取的信息字段。

## 三、功能清单

### 3.1 判决书解析功能

判决书解析的目标是**理解判决书说了什么**，为后续证据生成提供完整的信息输入。

#### 3.1.1 信息提取原则

**由证据生成的需求决定提取哪些信息。**

| 证据类型 | 需要的信息 | 决定来源 |
|----------|------------|----------|
| 起诉状 | 当事人信息、诉讼请求、事实与理由 | F2.1 起诉状规划 |
| 合同 | 当事人、标的物、金额、日期 | F2.3 合同生成 |
| 附件 | 金额清单、物品清单 | F2.6 附件生成 |

> 不是判决书有什么就提取什么，而是**后面要生成什么，就提取什么**。

#### 3.1.2 功能定义

| 功能编号 | 功能名称 | 业务意义 | 功能范围 |
|----------|----------|----------|----------|
| F1.1 | 文本提取 | 判决书是PDF格式，需要提取为可处理的文本 | 1. 读取PDF文件<br>2. 提取文本内容<br>3. 保持段落结构 |
| F1.2 | 当事人识别 | 证据需要明确当事人身份信息 | 1. 识别原告名称<br>2. 识别被告名称<br>3. 提取当事人基本信息（名称、住所地、法定代表人等） |
| F1.3 | 案件事实提取 | 起诉状和证据都需要基于案件事实 | 1. 提取案件基本事实（合同签订时间、双方约定等）<br>2. 提取合同履行情况（支付、违约事实等）<br>3. 提取法院认定的事实 |
| F1.4 | 诉讼请求提取 | 起诉状需要完整包含诉讼请求 | 1. 提取本金诉求<br>2. 提取利息诉求<br>3. 提取违约金诉求<br>4. 提取其他诉求（诉讼费用等） |
| F1.5 | 关键金额提取 | 证据中的金额必须准确 | 1. 提取合同金额<br>2. 提取已付金额<br>3. 提取剩余欠款金额<br>4. 提取违约金金额 |

> **案件类型扩展说明**：本功能通过配置化设计支持不同案件类型。融资租赁案件提取"租金"、"租赁物"等字段；金融借款案件提取"贷款本金"、"还款计划"等字段；民间借贷案件提取"出借本金"、"借款利率"等字段。

### 3.2 证据生成功能

证据生成的目标是**根据判决书内容生成符合法院标准的证据材料**。

#### 3.2.1 证据规划（核心模块）

证据规划的目标是**通过判决书倒推原告提供了哪些证据**，并分析是否完整。

| 功能编号 | 功能名称 | 业务意义 | 功能范围 |
|----------|----------|----------|----------|
| F2.1 | 案情分析 | 理解案件的基本事实 | 1. 分析合同签订情况<br>2. 分析合同履行情况<br>3. 分析违约事实<br>4. 分析法院认定的事实 |
| F2.2 | 诉求提取 | 从判决书中提取原告的诉讼请求 | 1. 提取本金诉求<br>2. 提取利息诉求<br>3. 提取违约金诉求<br>4. 提取其他诉求 |
| F2.3 | 证据清单倒推 | 从判决书倒推原告提供了哪些证据 | 1. 从判决书提取原告证据清单<br>2. 提取每个证据的名称和内容摘要<br>3. 标注法院是否采纳该证据 |
| F2.4 | 证据完整性分析 | 分析原告证据是否完整，是否有缺失 | 1. 对照诉求与证据清单<br>2. 识别缺失的证据<br>3. 规划需要补充的证据 |
| F2.5 | 证据规划输出 | 生成完整的证据生成计划 | 1. 列出需要生成的证据清单<br>2. 标注每份证据的数据来源<br>3. 标注证据与诉求的对应关系 |

> **核心逻辑**：不是"根据案件类型生成证据"，而是"通过判决书倒推原告提供了什么证据"。

| 输入 | 处理 | 输出 |
|------|------|------|
| 判决书 | 分析案情 + 提取诉求 + 倒推证据清单 + 完整性分析 | 证据规划（需要生成哪些证据，数据来源是什么） |

#### 3.2.2 证据生成

| 功能编号 | 功能名称 | 业务意义 | 功能范围 |
|----------|----------|----------|----------|
| F2.6 | 起诉状生成 | 生成符合法院标准的民事起诉状 | 1. 生成当事人信息<br>2. 生成诉讼请求（与判决书一致）<br>3. 生成事实与理由<br>4. 生成证据清单<br>5. 生成落款 |
| F2.7 | 合同生成 | 合同是金融案件的核心证据 | 1. 生成合同基本信息<br>2. 生成标的物描述<br>3. 生成价款条款<br>4. 生成权利义务条款<br>5. 生成违约责任条款<br>6. 生成签署栏 |
| F2.8 | 凭证生成 | 银行凭证证明资金往来事实 | 1. 生成付款凭证<br>2. 生成收款凭证 |
| F2.9 | 文书生成 | 公证书、裁定书等是辅助证据 | 1. 生成公证书摘要<br>2. 生成裁定书摘要 |
| F2.10 | 附件生成 | 附件是合同/凭证的组成部分 | 1. 生成附件清单<br>2. 生成还款/支付计划 |
| F2.11 | 证据索引生成 | 证据索引便于法官查阅证据 | 1. 按证据组分类<br>2. 列出每个证据的名称、类型、证明目的<br>3. 生成证据与诉求的关联关系 |

> **案件类型扩展说明**：
> - 融资租赁案件：合同类型=融资租赁合同，附件=租赁物清单、租金支付计划
> - 金融借款案件：合同类型=借款合同，附件=还款计划、抵押物清单
> - 民间借贷案件：合同类型=借条/借款合同，附件=还款记录
> - 保理合同案件：合同类型=保理合同，附件=应收账款清单、转让通知
> - 新案件类型只需配置证据类型和附件类型，无需修改代码

### 3.3 PDF输出功能

PDF输出的目标是**将生成的证据材料渲染为符合法院标准的PDF文件**。

| 功能编号 | 功能名称 | 业务意义 | 功能范围 |
|----------|----------|----------|----------|
| F3.1 | 文本渲染 | 证据内容需要以PDF格式呈现 | 1. 渲染标题<br>2. 渲染正文段落<br>3. 渲染落款 |
| F3.2 | 表格渲染 | 附件中的表格需要规范化呈现 | 1. 渲染表格表头<br>2. 渲染数据行<br>3. 渲染合计行（如有） |
| F3.3 | 分页处理 | 每份证据需要独立成页便于查阅 | 1. 起诉状单独成页<br>2. 每份证据独立成页<br>3. 每份附件独立成页 |
| F3.4 | 证据集打包 | 最终输出为单一PDF文件便于提交 | 1. 打包所有证据<br>2. 生成目录页<br>3. 生成页码 |

## 四、质量标准

### 4.1 内容质量

| 标准 | 要求 | 业务意义 |
|------|------|----------|
| 真实性 | 证据内容必须与判决书一致 | 证据必须真实反映案件情况 |
| 完整性 | 判决书中涉及的证据都必须生成 | 避免遗漏重要证据 |
| 准确性 | 当事人信息、金额、日期必须准确 | 错误信息会影响证据效力 |

### 4.2 格式质量

| 标准 | 要求 | 业务意义 |
|------|------|----------|
| 脱敏标记 | 证据中不允许出现脱敏标记 | 脱敏标记会导致证据无效 |
| 格式规范 | 符合法院对证据材料的标准格式要求 | 格式不规范可能被要求重新整理 |
| 证据编号 | 证据编号必须唯一，编号顺序合理 | 便于法官查阅和引用 |

### 4.3 数据一致性

| 检查项 | 要求 | 业务意义 |
|--------|------|----------|
| 金额一致 | 合同金额与附件中的金额合计必须一致 | 金额矛盾会影响证据可信度 |
| 当事人一致 | 同一当事人的名称在所有证据中必须保持一致 | 名称不一致会增加法院核实成本 |
| 日期一致 | 同一事件的日期在所有证据中必须保持一致 | 日期矛盾会影响事实认定 |

## 五、验收标准

### 5.1 功能验收

| 验收项 | 标准 |
|--------|------|
| 输入支持 | 系统能够识别并读取PDF格式的判决书文件 |
| 输出完整 | 系统输出包含民事起诉状和判决书中涉及的所有证据材料 |
| 格式正确 | 系统输出为符合法院标准的PDF文件，可直接用于提交 |
| 类型扩展 | 支持通过配置扩展新的案件类型 |

### 5.2 质量验收

| 验收项 | 标准 |
|--------|------|
| 无脱敏标记 | 证据材料中不允许出现"某某公司"等脱敏标记 |
| 内容准确 | 证据内容与判决书记载的事实保持一致 |
| 格式规范 | 证据材料格式符合法院提交材料的标准要求 |

## 六、约束条件

### 6.1 设计约束

| 约束 | 说明 |
|------|------|
| 脱敏标记隔离 | 在LLM交互中不出现脱敏标记，只使用当事人真实名称 |
| 要素驱动 | 系统负责从判决书提取要素，内容生成由LLM完成 |
| 简洁指令 | 告诉LLM要生成什么，不告诉它怎么做 |

### 6.2 性能约束

| 约束 | 说明 |
|------|------|
| 响应时间 | 单次证据生成请求应在合理时间内返回结果 |
| 资源使用 | 系统运行资源消耗应在合理范围内 |

### 6.3 可扩展性约束

| 约束 | 说明 |
|------|------|
| 案件类型扩展 | 新增案件类型无需修改系统代码，只需添加配置文件 |
| 证据类型扩展 | 新增证据类型只需配置证据模板和字段映射 |
| 附件类型扩展 | 新增附件类型只需配置数据来源和渲染方式 |
