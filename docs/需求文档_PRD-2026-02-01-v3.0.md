# 金融案件证据集生成系统 v3.0 - 产品需求文档

**文档编号**: PRD-2026-02-01-v3.0
**版本**: v3.0.0
**日期**: 2026-02-01

---

## 一、产品概述

### 1.1 产品目标

输入一份判决书PDF，输出一份符合法院标准的证据集PDF。

### 1.2 用户故事

| 角色 | 场景 | 需求 |
|------|------|------|
| 法官 | 收到金融案件判决书 | 快速生成完整的证据提交材料 |
| 律师 | 代理金融案件 | 整理证据材料，生成标准化的证据集 |
| 当事人 | 需要提交证据材料 | 一键生成符合法院要求的证据文档 |

### 1.3 产品愿景

**告诉系统要什么，系统就生成什么。**

---

## 二、系统边界

### 2.1 输入输出

```
输入: 判决书 PDF
输出: 证据集 PDF（包含起诉状 + 证据材料）
```

### 2.2 功能范围

| 功能 | 说明 |
|------|------|
| 判决书解析 | 理解判决书内容，提取关键信息 |
| 证据生成 | 根据判决书内容生成证据材料 |
| PDF输出 | 生成符合法院标准的证据集 |

### 2.3 非功能范围（不包含）

- 判决书内容生成
- 法院系统对接
- 电子送达

---

## 三、业务流程

### 3.1 核心业务流程

```
判决书
   │
   ▼
F2.1 案情分析 ────→ 案情基本数据集（当事人、签约主体、标的物、标的金额等）
   │
   ▼
F2.2 诉求提取 ────→ 诉讼请求（本金、利息、违约金等）
   │
   ▼
F2.3 证据规划 ────→ 证据需求清单（根据诉求+案件性质规划）
   │
   ▼
F2.4 证据收集 ────→ 完整证据列表（判决书提取 + 自行编造）
   │
   ▼
F2.5 证据列表创建 ←核心：从案情基本数据集提取每份证据的关键信息
   │
   ▼
LLM 生成证据 ────→ 证据文本（只使用真实名称）
   │
   ▼
F2.6 起诉状生成
F2.7-F2.10 证据生成
F2.11 证据索引生成
   │
   ▼
F3 PDF输出 ────→ 证据集 PDF
```

### 3.2 数据关系说明

| 概念 | 本质 | 包含内容 |
|------|------|----------|
| **案情基本数据集** | F2.1案情分析的产物 | 当事人、签约主体、标的物、标的金额等关键要素 |
| **证据列表** | F2.5证据列表创建的产物 | 每份证据需要的关键信息（不可缺失的），支撑的诉求 |
| **LLM生成** | 根据证据列表生成 | 只使用真实名称，不使用脱敏标记 |

## 四、功能清单

### 4.1 证据规划模块（核心）

证据规划的目标是**像原告代理律师一样，先搞清诉求，再规划需要哪些证据来支撑诉求**。

| 功能编号 | 功能名称 | 业务意义 | 功能范围 |
|----------|----------|----------|----------|
| F2.1 | 案情分析 | 理解案件基本事实，生成**案情基本数据集** | 1. 分析合同签订情况<br>2. 分析合同履行情况<br>3. 分析违约事实<br>4. 分析法院认定的事实<br>5. 输出案情基本数据集 |
| F2.2 | 诉求提取 | 从判决书中提取原告的诉讼请求 | 1. 提取本金诉求<br>2. 提取利息诉求<br>3. 提取违约金诉求<br>4. 提取其他诉求 |
| F2.3 | 证据规划 | 根据诉求+案件性质，规划需要哪些证据来支撑诉求 | 1. 根据诉求确定需要证明的事实<br>2. 根据案件性质（合同类型）确定证据类型<br>3. 规划每项诉求需要哪些证据支撑<br>4. 输出证据需求清单 |
| F2.4 | 证据收集 | 从判决书提取 + 自行编造缺失但必须的证据 | 1. 从判决书提取原告证据及法院采纳情况<br>2. 对照证据需求，识别缺失的证据<br>3. 自行编造缺失但支撑诉求必须的证据<br>4. 输出完整证据列表（来源+是否编造） |
| F2.5 | 证据列表创建 | **从案情基本数据集提取每份证据需要的关键信息** | 1. 从案情基本数据集（F2.1输出）提取每份证据需要的**不可缺失的关键信息**<br>2. 为每份证据标注关键信息（当事人、金额、日期等）<br>3. 标注每份证据支撑的诉求<br>4. 输出证据列表（供LLM生成证据使用） |

#### 4.1.1 案情基本数据集（F2.1输出）

案情基本数据集是证据生成的基础数据，包含案件的核心要素：

| 要素类别 | 包含内容 |
|----------|----------|
| 当事人信息 | 原告名称、被告名称、住所地、法定代表人 |
| 签约主体信息 | 公司名称、统一社会信用代码、注册地址、法定代表人、银行账户 |
| 合同类型 | 融资租赁合同、借款合同、保理合同、担保合同等 |
| 标的物描述 | 租赁物名称/贷款金额/应收账款等 |
| 标的金额 | 合同金额、已付金额、剩余欠款、违约金金额 |
| 日期信息 | 签约日期、履行日期、违约日期、诉讼日期 |
| 其他关键信息 | 根据案件类型补充的要素 |

#### 4.1.2 证据列表（F2.5输出）

证据列表是LLM生成证据的依据，每份证据标注其关键信息：

| 证据类型 | 关键信息（不可缺失） | 支撑诉求 |
|----------|---------------------|----------|
| 合同类 | 当事人、标的物描述、价款条款、日期、签署栏 | 本金、租金 |
| 凭证类 | 金额、收付款方、日期、交易摘要 | 已付款项、欠款金额 |
| 文书类 | 文书类型、发证机关、关键内容、日期 | 权利证明 |
| 附件类 | 序号、名称、数量/金额（表格形式） | 标的明细 |

> **核心逻辑**：
> - 案情基本数据集 = F2.1的产物
> - 证据列表 = F2.5从案情基本数据集提取每份证据的关键信息
> - LLM根据证据列表生成证据，只使用真实名称，不使用脱敏标记

### 4.2 证据生成功能

| 功能编号 | 功能名称 | 业务意义 | 功能范围 |
|----------|----------|----------|----------|
| F2.6 | 起诉状生成 | 生成符合法院标准的民事起诉状 | 1. 生成当事人信息<br>2. 生成诉讼请求<br>3. 生成事实与理由<br>4. 生成证据清单<br>5. 生成落款 |
| F2.7 | 合同生成 | 合同是金融案件的核心证据 | 1. 生成合同基本信息<br>2. 生成标的物描述<br>3. 生成价款条款<br>4. 生成权利义务条款<br>5. 生成违约责任条款<br>6. 生成签署栏 |
| F2.8 | 凭证生成 | 银行凭证证明资金往来事实 | 1. 生成付款凭证<br>2. 生成收款凭证 |
| F2.9 | 文书生成 | 公证书、裁定书等是辅助证据 | 1. 生成公证书摘要<br>2. 生成裁定书摘要 |
| F2.10 | 附件生成 | 附件是合同/凭证的组成部分 | 1. 生成附件清单<br>2. 生成还款/支付计划 |
| F2.11 | 证据索引生成 | 证据索引便于法官查阅证据 | 1. 按证据组分类<br>2. 列出每个证据的名称、类型、证明目的<br>3. 生成证据与诉求的对应关系 |

### 4.3 PDF输出功能

| 功能编号 | 功能名称 | 业务意义 | 功能范围 |
|----------|----------|----------|----------|
| F3.1 | 文本渲染 | 证据内容需要以PDF格式呈现 | 1. 渲染标题<br>2. 渲染正文段落<br>3. 渲染落款 |
| F3.2 | 表格渲染 | 附件中的表格需要规范化呈现 | 1. 渲染表格表头<br>2. 渲染数据行<br>3. 渲染合计行（如有） |
| F3.3 | 分页处理 | 每份证据需要独立成页便于查阅 | 1. 起诉状单独成页<br>2. 每份证据独立成页<br>3. 每份附件独立成页 |
| F3.4 | 证据集打包 | 最终输出为单一PDF文件便于提交 | 1. 打包所有证据<br>2. 生成目录页<br>3. 生成页码 |

### 4.4 案件类型与证据类型映射

本系统通过配置化设计支持多种案件类型：

| 案件类型 | 需要生成的证据 |
|----------|----------------|
| 融资租赁纠纷 | 融资租赁合同、付款凭证、公证书、租赁物清单、租金支付计划 |
| 金融借款合同纠纷 | 借款合同、银行流水、还款计划、担保合同 |
| 民间借贷纠纷 | 借条、银行转账凭证、还款记录 |
| 保理合同纠纷 | 保理合同、应收账款清单、转让通知 |
| 担保合同纠纷 | 担保合同、抵押物清单、他项权利证明 |

> **扩展说明**：新增案件类型只需配置对应的证据类型列表；新增证据类型只需配置需要提取的关键信息。

## 五、质量标准

### 5.1 内容质量

| 标准 | 要求 | 业务意义 |
|------|------|----------|
| 真实性 | 证据内容必须与判决书一致 | 证据必须真实反映案件情况 |
| 完整性 | 判决书中涉及的证据都必须生成 | 避免遗漏重要证据 |
| 准确性 | 当事人信息、金额、日期必须准确 | 错误信息会影响证据效力 |

### 5.2 格式质量

| 标准 | 要求 | 业务意义 |
|------|------|----------|
| 脱敏标记 | 证据中不允许出现脱敏标记 | 脱敏标记会导致证据无效 |
| 格式规范 | 符合法院对证据材料的标准格式要求 | 格式不规范可能被要求重新整理 |
| 证据编号 | 证据编号必须唯一，编号顺序合理 | 便于法官查阅和引用 |

### 5.3 数据一致性

| 检查项 | 要求 | 业务意义 |
|--------|------|----------|
| 金额一致 | 合同金额与附件中的金额合计必须一致 | 金额矛盾会影响证据可信度 |
| 当事人一致 | 同一当事人的名称在所有证据中必须保持一致 | 名称不一致会增加法院核实成本 |
| 日期一致 | 同一事件的日期在所有证据中必须保持一致 | 日期矛盾会影响事实认定 |

## 六、验收标准

### 6.1 功能验收

| 验收项 | 标准 |
|--------|------|
| 输入支持 | 系统能够识别并读取PDF格式的判决书文件 |
| 输出完整 | 系统输出包含民事起诉状和判决书中涉及的所有证据材料 |
| 格式正确 | 系统输出为符合法院标准的PDF文件，可直接用于提交 |
| 类型扩展 | 支持通过配置扩展新的案件类型 |

### 6.2 质量验收

| 验收项 | 标准 |
|--------|------|
| 无脱敏标记 | 证据材料中不允许出现"某某公司"等脱敏标记 |
| 内容准确 | 证据内容与判决书记载的事实保持一致 |
| 格式规范 | 证据材料格式符合法院提交材料的标准要求 |

## 七、约束条件

### 7.1 设计约束

| 约束 | 说明 |
|------|------|
| 脱敏标记隔离 | 在LLM交互中不出现脱敏标记，只使用真实名称 |
| 要素驱动 | 系统负责从判决书提取要素，内容生成由LLM完成 |
| 简洁指令 | 告诉LLM要生成什么，不告诉它怎么做 |

### 7.2 性能约束

| 约束 | 说明 |
|------|------|
| 响应时间 | 单次证据生成请求应在合理时间内返回结果 |
| 资源使用 | 系统运行资源消耗应在合理范围内 |

### 7.3 可扩展性约束

| 约束 | 说明 |
|------|------|
| 案件类型扩展 | 新增案件类型无需修改系统代码，只需添加配置文件 |
| 证据类型扩展 | 新增证据类型只需配置证据模板和字段映射 |
| 附件类型扩展 | 新增附件类型只需配置数据来源和渲染方式 |
