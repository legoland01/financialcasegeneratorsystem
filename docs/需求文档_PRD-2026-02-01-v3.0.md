# 金融案件证据集生成系统 v3.0 - 产品需求文档

**文档编号**: PRD-2026-02-01-v3.0
**版本**: v3.0.0
**日期**: 2026-02-01

---

## 一、产品概述

### 1.1 产品目标

输入一份判决书PDF，输出一份符合法院标准的证据集PDF。

### 1.2 用户故事

| 角色 | 场景 | 需求 |
|------|------|------|
| 法官 | 收到融资租赁案件判决书 | 快速生成完整的证据提交材料 |
| 律师 | 代理融资租赁案件 | 整理证据材料，生成标准化的证据集 |
| 当事人 | 需要提交证据材料 | 一键生成符合法院要求的证据文档 |

### 1.3 产品愿景

**告诉系统要什么，系统就生成什么。**

---

## 二、系统边界

### 2.1 输入输出

```
输入: 判决书 PDF
输出: 证据集 PDF（包含起诉状 + 证据材料）
```

### 2.2 功能范围

| 功能 | 说明 |
|------|------|
| 判决书解析 | 理解判决书内容，提取关键信息 |
| 证据生成 | 根据判决书内容生成证据材料 |
| PDF输出 | 生成符合法院标准的证据集 |

### 2.3 非功能范围（不包含）

- 判决书内容生成
- 法院系统对接
- 电子送达

---

## 三、功能清单

### 3.1 判决书解析功能

| 功能编号 | 功能名称 | 说明 |
|----------|----------|------|
| F1.1 | 文本提取 | 从PDF判决书中提取可编辑的文本内容 |
| F1.2 | 当事人识别 | 从判决书正文中识别并提取案件当事人信息，包括名称、身份信息、地址等 |
| F1.3 | 事实提取 | 从判决书的事实认定部分提取案件的关键事实，包括合同签订、履行、违约等 |
| F1.4 | 诉讼请求提取 | 从判决书中提取原告的诉讼请求，包括本金、利息、违约金等金额诉求 |
| F1.5 | 金额提取 | 从判决书中提取所有与金额相关的信息，包括合同金额、租金、违约金等，并标注单位 |

### 3.2 证据生成功能

| 功能编号 | 功能名称 | 说明 |
|----------|----------|------|
| F2.1 | 起诉状生成 | 根据判决书中的诉讼请求，生成符合法院标准的民事起诉状，包含当事人信息、诉讼请求、事实与理由、落款等 |
| F2.2 | 合同生成 | 根据判决书涉及的合同，生成完整的合同文本，包含当事人信息、合同标的、价款、履行方式、违约责任等 |
| F2.3 | 凭证生成 | 根据判决书涉及的资金往来，生成相应的支付凭证或收据，包含金额、日期、收付款方信息 |
| F2.4 | 文书生成 | 根据判决书涉及的公证书、裁定书、判决书等，生成相应的文书文本或摘要 |
| F2.5 | 附件生成 | 根据合同内容生成相关附件，包括租赁物清单、租金支付计划、设备清单等表格类附件 |
| F2.6 | 证据索引生成 | 生成证据材料的总清单，按证据组分类，标注每个证据的名称、类型、证明目的 |

### 3.3 PDF输出功能

| 功能编号 | 功能名称 | 说明 |
|----------|----------|------|
| F3.1 | 文本渲染 | 将生成的证据文本渲染为标准PDF格式，包含标题、正文、落款等 |
| F3.2 | 表格渲染 | 将附件中的表格数据渲染为PDF表格，包含表头、数据行、合计行等 |
| F3.3 | 分页处理 | 对证据材料进行正确分页，每份独立证据从新页开始 |
| F3.4 | 证据集打包 | 将所有证据材料打包为单一PDF文件，包含目录页、证据正文、附件等 |

## 四、质量标准

### 4.1 内容质量

| 标准 | 要求 |
|------|------|
| 真实性 | 证据内容必须与判决书一致，不得虚构或篡改 |
| 完整性 | 判决书中涉及的证据材料都必须生成，不得遗漏 |
| 准确性 | 当事人信息、金额、日期等关键数据必须准确无误 |

### 4.2 格式质量

| 标准 | 要求 |
|------|------|
| 脱敏标记 | 证据中不允许出现脱敏标记（如"某某公司"、"某年某月某日"等） |
| 格式规范 | 符合法院对证据材料的标准格式要求，包括标题格式、正文格式、落款格式等 |
| 证据编号 | 证据编号必须唯一，编号顺序必须合理，便于查阅和引用 |

### 4.3 数据一致性

| 检查项 | 要求 |
|--------|------|
| 金额一致 | 合同金额与附件中的金额合计必须一致 |
| 当事人一致 | 同一当事人的名称在所有证据中必须保持一致 |
| 日期一致 | 同一事件的日期在所有证据中必须保持一致 |

## 五、验收标准

### 5.1 功能验收

| 验收项 | 标准 |
|--------|------|
| 输入支持 | 系统能够识别并读取PDF格式的判决书文件 |
| 输出完整 | 系统输出包含民事起诉状和判决书中涉及的所有证据材料 |
| 格式正确 | 系统输出为符合法院标准的PDF文件，可直接用于提交 |

### 5.2 质量验收

| 验收项 | 标准 |
|--------|------|
| 无脱敏标记 | 证据材料中不允许出现"某某公司"等脱敏标记 |
| 内容准确 | 证据内容与判决书记载的事实保持一致 |
| 格式规范 | 证据材料格式符合法院提交材料的标准要求 |

## 六、约束条件

### 6.1 设计约束

| 约束 | 说明 |
|------|------|
| 脱敏标记隔离 | 在LLM交互中不出现脱敏标记，只使用当事人真实名称 |
| 要素驱动 | 系统负责从判决书提取要素，内容生成由LLM完成 |
| 简洁指令 | 告诉LLM要生成什么，不告诉它怎么做 |

### 6.2 性能约束

| 约束 | 说明 |
|------|------|
| 响应时间 | 单次证据生成请求应在合理时间内返回结果 |
| 资源使用 | 系统运行资源消耗应在合理范围内 |

| 术语 | 定义 |
