# 金融案件测试数据生成系统方案

## 项目简介

本项目旨在设计一个控制大模型自动生成金融案件测试卷宗的核心逻辑和提示词框架。系统以公开的金融案件判决书（含脱敏内容）为输入，自动生成符合诉讼程序逻辑的完整测试卷宗。

## 背景

为开发金融案件计算软件，需要大量测试用卷宗材料。由于法院卷宗保密，需要基于公开发布的判决书，用大模型自动生成该案件的原始卷宗，以供测试使用。

## 核心目标

1. **基于判决书生成**：有充分依据，避免编造
2. **自动化脱敏处理**：将脱敏信息替换为真实感的模拟信息
3. **逻辑自洽**：确保所有文件之间数据一致
4. **质量可控**：通过多层次质量检查减少人工干预
5. **通用性强**：适配不同类型的金融案件

## 整体架构

```
判决书（含脱敏信息）
    ↓
阶段0：判决书解析与全局规划
    ├─ 0.1 结构化提取
    ├─ 0.2 脱敏替换策划
    ├─ 0.3 交易结构重构
    ├─ 0.4 关键数字提取
    └─ 0.5 证据归属规划
    ↓
阶段1：原告起诉包生成（独立）
    ├─ 1.1 民事起诉状
    ├─ 1.2 原告证据包
    └─ 1.3 原告程序性文件
    ↓
阶段2：被告答辩包生成（独立）
    ├─ 2.1 民事答辩状
    ├─ 2.2 被告证据包
    └─ 2.3 被告程序性文件
    ↓
阶段3：法院审理包生成（引用前两阶段）
    ├─ 3.1 庭审笔录
    └─ 3.2 判决书脱敏替换
    ↓
输出：完整测试卷宗
```

## 第一版scope

- **支持案件类型**：融资租赁合同纠纷
- **证据包特点**：
  - 原告证据包：支持原告诉讼请求的完整文件
  - 被告证据包：支持被告抗辩意见的完整文件，与原告包相对独立
  - 法院采信证据包：仅从判决书映射到前两包，不生成新文件
- **质量控制**：半自动化，关键节点可能需要人工审核

## 使用说明

1. 阅读 `docs/00_方案概述.md` 了解整体设计思路
2. 阅读 `docs/01_总体架构设计.md` 了解三阶段流程
3. 根据 `docs/02_阶段0详细设计.md` 理解判决书解析逻辑
4. 参考 `prompts/` 目录下的提示词模板进行开发
5. 使用 `quality_control/` 下的规则进行质量控制
